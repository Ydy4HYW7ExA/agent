# Term Governance

术语治理最容易被低估，因为“换个词”看上去像是表面修辞。真正发生的事远比修辞深。文档、工作流和代码注释之所以能彼此指向同一对象，靠的是共享主表述。主表述一旦漂移，链接还在，语义却开始松动，读者会越来越难判断这些文件究竟是在说一件事还是几件事。

共享总表保持收紧状态。收紧不等于只收跨 skill 正文反复出现的词。有些对象虽然不穿过很多 skill 的正文，却已经成为系统级入口对象。这类词只有在它真的已经站成入口时才值得进入总表；过早升格只会让共享总表重新长成一个混杂的总目录。

局部术语表让每个 skill 继续维护自己的内部边界，共享总表则负责入口层的统一。两者不是竞争关系，而是两种不同粒度的入口。总表让读者先看骨架，局部表让读者再进内部。

术语迁移还要有可执行格式。没有格式时，每次迁移都要重新发明记录方法，读者很难扫描哪些词已经退场、哪些词仍是正式主词、哪些词只是为了兼容检索而临时保留。格式一旦固定，迁移的结构成本会大幅下降。

可以用两个最小例子看出“什么时候升格、什么时候不升格”。`clean expression` 值得进入共享总表，因为它跨越对话清洗、文档写作和后续任务入口。`package language` 当前不进入共享总表，因为它仍主要停留在 `repository-construction` 的局部边界内。前一个词让多个 skill 共享同一个入口对象；后一个词目前仍由单个 skill 的局部真值承接。升格判断看共享入口是否已经出现，不看单个对象是否“听起来重要”。

迁移也可以用最小例子说明。若一个局部词被判定会误导结构，动作不是“在正文里到处替换一下”，而是先把新词写进术语表，再把旧词降格为 `alias` 或 `deprecated term`，最后才同步关键入口。这样，迁移的主线始终是“术语表 -> 入口文件 -> 解释与例子”，而不是“搜索替换 -> 之后再补说明”。
