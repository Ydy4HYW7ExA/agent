# Package System Evolution

仓库的包体系不会一直停在“新增一个包”这个动作上。仓库开始承接持续变化的需求后，活包集合会跟着变化：有的包出生，有的包扩展，有的包开始分裂，有的包进入迁移，有的包被清理。真正需要被维护的，不只是单个包的结构，还包括仓库根里“当前有哪些包仍然活着”这件事。

包体系演化时，最重要的对象是 `active package set`。这组包对象给出当前仓库真正承接需求的边界图。新增包时，它增长；拆分包时，它先扩张再收缩；退役包时，它收缩。很多结构讨论看上去在谈目录，实际都在谈这组对象该怎么变。

拆分是包体系演化里最容易失真的地方。一个包出现拆分信号，不表示它应当立刻删除；它表示某个 `business/exit` 已经直接承接了多个 `business` 非出口文件，这个包不适合继续把这些边界收在同一个活包里。更稳的动作是：先把新包建出来，迁移消费者，再把旧包降到 `migrating`，最后再进入 `retiring`。这样，读者在任何时点都能知道当前包体系到底处在什么状态。

若包体系变化同时来自运行时关系重组，还要再看一层：当前 `runtime triad` 有没有变化，两侧 `instance package` 有没有变化，原来的最小业务域包群是不是已经换了一种组合。只有把关系抽象层和实例层一起写实，包体系演化才不会把“结构变化”误写成“过程变化”，或反过来把过程迁移误写成结构重组。

合并和清理也一样。两个包若长期承接同一问题边界，而且统一语言和固定出入口集合正在重复，就值得考虑合并；某个包若已经不再承接活需求，只剩历史出口或迁移残留，就值得进入退役阶段。关键不在“目录还在不在”，而在包体系状态工件有没有把这些变化写实。

包体系演化至少留下三类工件：

```json
{
  "package_system_artifacts": [
    "temp/architecture/repository-root/active-packages.json",
    "temp/packages/<package-name>/structure.json",
    "temp/packages/<package-name>/migration-notes.md"
  ]
}
```

第一类工件承接仓库根层的活包集合，第二类工件承接单包结构，第三类工件承接迁移过程。三者一起出现，仓库的包体系演化才会真正可追踪、可核对、可清理。

仓库根层的活包集合工件不只是一张包名清单，它还是当前阶段工件。最小对象应当同时包含：

```json
{
  "repository_stage": "forming | expanding | restructuring | stabilizing | shrinking",
  "active_packages": [],
  "migrating_packages": [],
  "retiring_packages": []
}
```

这样，读者在任何时点都能同时回答两件事：当前有哪些包仍承接活需求，当前包体系整体正处在什么阶段。若只写包名列表，不写阶段，迁移中的动作就会失去全局位置。
