# Conversation Window

把 `raw material` 理解成“当前轮对话”，很多仍有决定力的材料就会被漏掉。很多任务并不是一轮成形的。用户可能先在前几轮里暴露焦虑、约束和反例，到后面几轮才说出自己真正想收的结构。如果只盯当前轮，早先那些仍未正式化的材料就会掉到定义之外。

更稳的做法是引入任务窗口。窗口的边界由任务决定，不由消息轮次决定。只要一段材料仍然没有被清洗并固化到任务、术语或长期文档里，它就仍属于当前窗口内的原始材料。只要某段内容已经被正式化，它就从 `raw material` 移出，后续由对应 skill 接管。

窗口如果只停在脑中，协作时会很快失真。更稳的做法是：一旦窗口材料需要被跨轮保存或被多人反复引用，就把它写成草稿工件，放进目标仓库自己的 `temp/`。当前仓库里的默认落点如下：

```json
{
  "task_window_persistence": {
    "task_scoped": "temp/tasks/",
    "architecture_scoped": "temp/architecture/",
    "skill_scoped": "temp/skills/"
  }
}
```

窗口材料进入正式文档后，应从草稿区移出或删除引用。这样，窗口始终只承接未正式化内容。
