# Documentation Authoring Writing

信息落点先按对象分，再按读者决策分。方法写进 skill 文档，仓库事实写进仓库文档，代码局部边界写进注释，临时分析材料写进草稿工件。若一条信息不能稳定落在这四者之一，说明它还不够稳定，不该急着写成长文档。

共享术语还要有主落点。一个词已经在术语表里拥有主定义后，其他文件就不再重复写完整定义，而是做最小引用或最小释义，并回链主落点。入口文件尤其要克制，避免根入口、README 和局部文档各自长出一套定义。

术语首现遵守一条硬规则：先给对象定义，再继续使用这个词。定义说清三件事：它指什么，它不指什么或它和相邻对象怎样分开，它在当前文档里承担什么作用。只给名字不给定义，读者会被迫一边读一边猜对象；猜测一旦发生，文档的推导就已经松掉了。

把这条规则压成最小检查对象，就是：

```json
{
  "first_use_check": {
    "term_defined": true,
    "boundary_stated": true,
    "role_in_current_doc_stated": true
  }
}
```

正文与结构化元素的配合要明确：

| 结构 | 适合描述什么 |
| --- | --- |
| 自然段 | 对象、边界、推导、后果 |
| `tree` | 目录结构、文件结构 |
| JSON 片段 | 状态模型、决策输入输出、规则对象 |
| 表格 | 对比、落点、职责分工 |
| 关系块 | `A -> B` 这类依赖或流向 |

措辞也要收紧。正文优先使用常用词，直接说对象、条件、后果和责任。下面这些写法应主动避开：

| 类型 | 避开什么 | 更稳的写法 |
| --- | --- | --- |
| 黑话 | 作者自造的包装词、只有体系内部才懂的缩略说法 | 常用词或先定义后的术语 |
| 防御式转折 | 连续使用“不是……而是……”先设靶子再转身说明 | 直接陈述条件、原因、结果 |
| 作者手势 | “本文件处理的是……”“它回答的是……”反复出现 | 先写对象，再自然展开 |

一个最小对照例子如下：

| 较差写法 | 更稳的写法 |
| --- | --- |
| “包不是目录，而是某种抽象概念。” | “`bounded context` 指围绕同一问题边界和统一语言成立的工作范围。包在这里按独立 `bounded context` 理解。” |
| “本文件处理的是包结构。” | “包结构分成 `core structure` 和 `optional surfaces` 两层。” |

文档只要在描述“结构”，就不该只停留在概念句上。补至少一种可以直接扫描的结构工件。结构工件是文档正文的一部分。

仓库根文档和 skill 文档也要分开看。根 README 适合写“仓库承接什么、顶层路径各承接什么”；它不适合重复共享术语总表，也不适合替 skill 重写内部 reference。skill reference 适合写“这个对象是什么、边界是什么、固定判断条件是什么”；它不适合讲阅读顺序，也不适合承担根入口导航。

把这个区别压成最小对照，就是：

| 文档面 | 应写什么 | 不应写什么 |
| --- | --- | --- |
| 根 `docs/README.md` | 仓库定位、顶层结构、路径职责、阅读顺序 | 共享术语总表、局部判断条件 |
| `SKILL.md` | skill 入口、最短路径、文件导航、使用约束 | 详细术语表、完整 workflow 步骤 |
| `references/*.md` | 静态真值、定义、固定判断条件 | 阅读顺序、作者手势 |
| `workflows/*.md` | 步骤、输入输出、核对方式 | 术语主定义 |

可以用下面这个对象检查一篇文档是否守住了写作纪律：

```json
{
  "definition_first": true,
  "single_source_for_shared_terms": true,
  "direct_language": true,
  "defensive_turns_removed": true,
  "structure_artifacts_present_when_needed": true
}
```
