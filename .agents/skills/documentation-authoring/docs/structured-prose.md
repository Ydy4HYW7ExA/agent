# Structured Prose

“高度结构化但不能把信息切碎”这句话如果只写成口号，没有用。真正要做的是给它一个可操作的写法。自然段负责承载完整判断，因为只有自然段能把对象、边界、推导和后果放在同一条语义线上。结构化元素负责把这条判断压实，让目录、状态、依赖、输入输出和落点变得可以直接扫描。二者各守其责，文档才会既能读也能用。

很多文档之所以让人不耐烦，原因在于信息总在错误的载体里。对象明明需要展开，却被拆成一串短句；目录结构明明适合用 `tree` 表示，却被埋在正文；决策输入输出明明适合用 JSON 或表格表示，却被散落在几个段落里。读者于是要一边读，一边自己重组信息。

更稳的办法，是先判断对象最适合哪种载体，再安排配合顺序。通常顺序是：场景引入 -> 自然段展开 -> 结构化元素压实 -> 回到自然段收束后果。这样，结构不会喧宾夺主，自然段也不会因为承担太多硬边界而变得发虚。

这套顺序还有一个前提：文中的词不能跑在定义前面。一个词第一次出现时，读者要立刻知道它指的对象、它和相邻对象的边界、它在当前文档里的作用。只有这一步成立，后面的自然段和结构化元素才真正有地面。词先跑，定义后补，读者就会在最关键的几段里持续猜测对象。

例如，写结构时至少应同时出现这两类工件：

```text
package-root/
├── config/
├── src/
└── docs/        # optional
```

```json
{
  "package_root": "packages/example",
  "implementation_variants": [
    {"language": "py", "suffix": "py"},
    {"language": "ts", "suffix": "ts"}
  ],
  "fixed_point_sets": {
    "business": {
      "exit": ["src/business/exit/open_proxy.py", "src/business/exit/open_proxy.ts"]
    }
  },
  "with_wrappers": false,
  "with_tests": false
}
```

前一个工件负责给出外形，后一个工件负责给出运行时决策。正文则负责解释为什么是这棵树、为什么是这些字段、为什么某些附属面是可选的。只剩工件，文档会变成图例；只剩正文，文档会失去扫描能力。三者配合，结构才真的站住。

还有一条容易被忽略的纪律：读者应该先看到对象，再看到作者的组织动作。写作里一旦反复出现“本文件处理的是”“它回答的是”“不是……而是……”这类句法，读者就会不断意识到作者在摆框架，而不是在接触对象本身。更稳的写法是直接陈述对象、条件和结果，让结构退到后面服务内容。

如果把一个“较差文稿”压成更稳的当前态，最小变化通常长这样：

| 较差段落 | 收束后的段落 |
| --- | --- |
| “本文件处理的是仓库结构。本文件的重点不是目录，而是结构对象。仓库不是随便放东西的地方，而是某种方法空间。” | “仓库根在这里承接协作规则、仓库事实、方法对象入口和仓库维护工具。顶层路径先按 `AGENTS.md`、`docs/`、`.agents/skills/`、`scripts/`、`temp/` 分开，再分别落成各自职责。” |

前一个写法不断提醒读者“作者正在组织内容”；后一个写法直接把对象和判断摆出来。读者先接触对象，再接触结构，最后才意识到作者做了组织动作。这个顺序一旦成立，文档的厚度就会变成帮助，而不是噪音。
