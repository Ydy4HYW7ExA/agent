# Skill System

把 skill 当成目录来理解，会天然低估它；把 skill 当成工具来理解，很多结构判断就自然了。目录只承接存放关系，工具承接使用关系。一个真正的工具必须回答三个问题：什么时候进入它，进入后先走哪条路径，用完后把读者送到哪里。skill 之所以需要被单独建设，原因在于这三个问题不会自动从“文件放在一起”里长出来。

这也是为什么“最小目录结构”和“最小可用状态”必须分开。前者回答的是：至少要有哪些位置，读者才知道结构没有塌；后者回答的是：至少要有哪些真实内容，读者才真的能用它。一个 skill 可以暂时没有 `docs/` 或 `scripts/`；它不能没有真实的入口、真值和路径。

脚手架脚本也按这个视角来理解。脚手架做的事，只是帮你免掉重复建目录和建空文件的劳动；它不替你决定 skill 的对象边界，不替你写出场景引入，也不替你安排相邻 skill。只要脚手架看上去像“已经拥有完整方法”，它制造的就是入口幻觉。

这个视角还会影响仓库级完整性怎么维护。当前仓库不保留根完整性检查脚本，因为根 `scripts/` 的职责已经收紧到仓库卫生。相应地，入口完整性、导航回链和结构收束改由 `skill-authoring` 的人工 workflow 承接。这样，检查动作仍然存在，但它被放回方法对象自己的边界内，而不是伪装成另一个根维护脚本。

可以用下面这个对象来区别三个阶段：

```json
{
  "scaffold_only": "directories + stubs",
  "minimum_structure": ["SKILL.md", "references/", "workflows/"],
  "minimum_viable_skill": [
    "real SKILL.md",
    "at least one real reference",
    "at least one real workflow"
  ]
}
```

入口弱化比内容不够更危险。skill 的内容可以继续长；一旦 `SKILL.md` 不能把读者送进 reference、workflow 和必要解释，后面的文件越多，读者越难判断这是不是同一个方法对象。
